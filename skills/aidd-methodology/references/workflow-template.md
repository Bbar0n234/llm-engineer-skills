# Шаблон рабочего процесса

Обкатанный шаблон workflow.md для проектов по AIDD-методологии. Адаптируй под конкретный проект: добавляй скоупы, убирай неактуальные разделы, дополняй соглашениями команды.

---

## Основной подход

AI-driven / Spec-driven development: разработчик выступает в роли архитектора и технического лида, делегируя написание кода LLM агенту. Успех достигается за счёт качественной документации — она является основой для эффективной работы агента.

## Двухуровневая система планирования

### Продуктовый план (project-plan.md)

Бизнес-ориентированный документ с задачами (TASK-001, TASK-002...), понятными заказчику и PM. Отражает что нужно реализовать с точки зрения продукта.

### Итерационная реализация (tasklist-\<scope\>.md)

Техническое планирование для разработчиков. Работа организована в итерации, которые:
- Выполняются параллельно в разных областях (backend, frontend, devops)
- Группируются в tasklist'ы по областям ответственности для удобства
- Могут закрывать одну или несколько задач из продуктового плана
- Имеют собственную нумерацию внутри области (feat-001, feat-002...)

Итерации из разных tasklist'ов в конечном счёте закрывают все задачи из project-plan.md.

## Ключевые понятия

### Задача (TASK)

Единица планирования в [project-plan.md](project/project-plan.md).

- Формат: TASK-001, TASK-002...
- Ориентирована на бизнес и планирование работ
- Используется для отслеживания прогресса на уровне проекта

### Итерация

Единица работы разработчика в `tasklist-<scope>.md`.

- Формат: `<type>-<number>` (например, feat-001, fix-002, refactor-003)
- Типы (аналогично Conventional Commits):
  - `feat` — новая функциональность
  - `fix` — исправление
  - `refactor` — рефакторинг
  - `docs` — документация
  - `test` — тесты
  - `chore` — инфраструктура, зависимости
  - `style` — форматирование кода
- Нумерация отдельная для каждого scope (001, 002... в рамках одного tasklist)
- Может закрывать одну или несколько задач (или их части)
- Порядок итераций не обязан совпадать с порядком задач в project-plan.md

### Связь задач и итераций

- Итерации в конечном счёте закрывают все задачи из project-plan.md
- Связь указывается в tasklist рядом с каждой итерацией
- project-plan.md синхронизируется по мере реализации итераций


### Соглашения по наименованию

Каждая итерация — отдельная директория: `<type>-<number>-<description>/`

Файлы внутри:
- `plan.md` — implementation plan
- `summary.md` — post-implementation summary

Пример:
```
feat-001-setup-infrastructure/
├── plan.md
└── summary.md
```

## Формат tasklist-\<scope\>.md

### Scope

Область ответственности: backend, frontend, devops, bot и т.д.

- Деление гибкое: можно выделить отдельный scope, если tasklist становится слишком большим
- Разные scope могут вести разные разработчики

### Легенда статусов

Единая система для всех tasklist'ов:
- 📋 Planned — Запланирован
- 🚧 In Progress — В работе
- ✅ Done — Завершен
- ⏸️ Paused — Приостановлен
- ❌ Cancelled — Отменен

### Overview

В начале файла — таблица для быстрого понимания прогресса:

| Итерация | Статус | Закрывает задачи |
|----------|--------|------------------|
| feat-001 | ✅ Done | TASK-001 |
| feat-002 | 🚧 In Progress | TASK-002, TASK-003 |
| fix-003 | 📋 Planned | TASK-004 |

### Список итераций

Каждая итерация содержит:
- **Цель** — краткое описание желаемого результата итерации
- **Статус выполнения** — текущее состояние
- **Связанные задачи** из project-plan.md
- **Описание состава работ** — детальная разбивка на подзадачи с чекбоксами
- **Проверка** — критерии приёмки и тесты
- **Артефакты** (Deliverables) — созданные файлы и документы
- **Индексация документации** — ссылки на всю связанную документацию

### Индексация документации

К каждой завершённой итерации добавляются ссылки на всю связанную документацию:

- **Implementation plan** — план реализации (`.plan.md`)
- **Post-implementation summary** — итоги реализации (`.summary.md`)
- **ADR** — принятые архитектурные решения
- **Техническая документация** — описания компонентов, API, инфраструктуры и т.д.
- **Прочая документация** — любые другие документы, появившиеся в ходе итерации

Пример:

```markdown
## Итерации

### feat-001: Настройка инфраструктуры

**Цель:** Создать основу для разработки — Docker окружение, CI/CD pipeline и базовую документацию.

- [x] Завершено
- Задачи: TASK-001
- Документация:
  - [План](iterations/backend/feat-001-setup-infra/plan.md)
  - [Summary](iterations/backend/feat-001-setup-infra/summary.md)
  - [ADR-001: Docker Compose](../adr/ADR-001-docker-compose.md)
  - [Описание инфраструктуры](../tech/infrastructure.md)

### feat-002: Основной функционал

**Цель:** Реализовать ключевую функциональность продукта согласно vision.md.

- [ ] В работе
- Задачи: TASK-002, TASK-003
- Документация:
  - [План](iterations/backend/feat-002-core-feature/plan.md)
```

## Рабочий процесс разработчика

### 1. Планирование итерации

- Выбрать задачи из project-plan.md для реализации
- Создать запись итерации в `tasklist-<scope>.md`
- Опционально: создать implementation plan в `/doc/tasks/iterations/<scope>/<type>-<number>-<desc>/plan.md`
  - План может быть в виде отдельного `plan.md` файла или в Plan Mode агента (история сообщений)
  - Если план не оформлен как отдельный файл — индексировать его не нужно

### 2. Реализация

- Работа с LLM агентом по плану
- Итеративное улучшение до достижения результата

### 3. Завершение итерации

- Создать post-implementation summary (`.summary.md`):
  - Отклонения от плана реализации
  - Принятые решения
  - Проблемы и их решения
  - Важные нюансы
- Актуализировать всю связанную документацию с помощью LLM-агента
  - **Обязательно:** отсмотреть вручную все изменения в документах и созданные документы
- Создать или обновить техническую документацию в `/doc/tech/`
- Создать ADR, если были приняты архитектурные решения
- Проиндексировать всю документацию в записи итерации в tasklist
- Обновить статус задач в project-plan.md

### Низкоуровневые детали

Не регламентируются — разработчик сам решает:
- Как декомпозировать итерацию на подзадачи
- Как вести промежуточные заметки
- Какие инструменты использовать
- Реализовать итерацию за один раз или за несколько подходов

## Отслеживание прогресса

### Для разработчика

`tasklist-<scope>.md` — детальный прогресс по итерациям с индексацией документации.

### Для НЕ разработчика (PM, заказчик)

`project-plan.md` — прогресс по задачам бизнес-уровня. Статусы задач синхронизируются по мере завершения итераций.

Для погружения в детали: project-plan.md → tasklist → связанная документация (план, summary, ADR, техническая документация). Это позволяет посмотреть принятые решения, нюансы реализации, отклонения от плана.

## Принципы

- Документация всегда актуальна
- Каждое архитектурное решение фиксируется в ADR
- Implementation plan — основа для работы LLM агента
- Вся документация итерации проиндексирована в tasklist
- Гибкость в деталях, системность в структуре
